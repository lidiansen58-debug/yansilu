# NotebookLM 辅助论文工作流草案 V1

## 1. 这份草案要解决什么

这份文档不是在定义“如何做一个 NotebookLM 连接器”。

它要解决的是一个更重要的问题：

如何帮助用户把 `借助 NotebookLM 加速读论文` 这件事，顺利推进为：

`论文来源 -> 文献笔记 -> 原创笔记 -> 关联 -> 洞见`

也就是说，NotebookLM 在这里不是终点，而是一个加速阅读与初步整理的中间层。

## 2. 核心产品判断

### 2.1 NotebookLM 不等于原创笔记

NotebookLM 输出的 summary、Q&A、study guide、notes，本质上都仍然属于“外部整理结果”。

它们可以帮助用户：

1. 更快理解论文结构。
2. 更快抓住核心论点、方法、结果和局限性。
3. 更快积累可加工的文献笔记候选。

但它们不应被直接视为：

1. 用户已经形成的判断。
2. 用户已经拥有的原创笔记。
3. 产品可以默认写入的永久观点。

### 2.2 真正要优化的不是“导入”，而是“转换”

NotebookLM 用户真正的痛点通常不是：

- 如何把内容搬进来。

而是：

- 如何不止停留在摘要和摘录。
- 如何把理解推进为判断。
- 如何把判断推进为可连接、可复用的原创笔记。

所以这条工作流的重点，不应是“导入成功”，而应是“完成转换”。

## 3. NotebookLM 在产品中的角色

NotebookLM 在研思录中的角色建议定义为：

1. 外部研究整理器。
2. 阅读加速器。
3. 文献笔记候选来源。

不建议定义为：

1. 主内容存储。
2. 原创笔记来源。
3. 双向同步对象。
4. 可直接替代用户判断的智能代理。

## 4. 目标用户

这套工作流主要服务以下用户：

1. 会读论文，但容易停在“看懂了、记不住”的用户。
2. 会使用 NotebookLM 辅助阅读，但缺少后续沉淀机制的用户。
3. 希望把论文阅读沉淀成原创判断，并继续连接出研究主题和写作结构的用户。

## 5. 目标链路

理想链路应是：

`论文来源 -> NotebookLM 辅助整理 -> 文献笔记候选 -> 用户转述 -> 原创笔记候选 -> 原创笔记 -> 关联与冲突 -> 洞见`

其中最关键的产品动作是：

1. 把 NotebookLM 输出拆成小颗粒候选。
2. 强制用户做自己的转述与判断。
3. 只在用户完成转述后，才允许生成原创笔记候选。

## 6. 工作流设计

### 6.1 Step 1：建立论文来源 Source

用户先建立一个论文来源对象，至少包括：

1. 论文标题。
2. 作者。
3. 年份。
4. DOI / arXiv / URL。
5. PDF 或文件附件。
6. 可选的 NotebookLM 辅助内容。

这一步的目的是让后续所有笔记都围绕同一个 `Source` 生长，而不是让 NotebookLM 文本成为孤立条目。

### 6.2 Step 2：导入 NotebookLM 辅助内容

初期不建议先做重型 API 直连，而建议优先支持以下轻入口：

1. 粘贴 NotebookLM summary。
2. 粘贴 NotebookLM Q&A。
3. 粘贴 NotebookLM study guide。
4. 粘贴 NotebookLM notes。

未来可扩展：

1. Public Notebook 链接导入。
2. 半自动页面采集。

但这些都应后于文本粘贴导入。

### 6.3 Step 3：拆成文献笔记候选

系统不应把整段 NotebookLM 文本当成一条笔记写入。

而应拆成多个小颗粒候选，例如：

1. 核心论点。
2. 方法摘要。
3. 关键实验结果。
4. 局限性。
5. 与研究问题相关的问答。
6. 可追溯引用片段。

这些对象默认都属于：

- `Literature Note candidates`

而不是 `Permanent Note candidates`。

### 6.4 Step 4：用户转述与判断

这是整个工作流最关键的一步。

系统应该要求用户补充自己的字段，而不是只保存 NotebookLM 文字。

每条候选建议用户至少回答这些问题中的一部分：

1. 我怎么理解这点？
2. 这和我正在研究/写作的问题有什么关系？
3. 我同意它什么，不同意它什么？
4. 这条结论成立的条件是什么？
5. 这和我已有哪条原创笔记有关？
6. 它最值得保留的不是信息，而是哪一个判断？

产品约束：

1. 如果没有用户自己的转述，不应把这条候选视为“已完成文献笔记”。
2. 如果只有 NotebookLM 改写结果，没有用户自己的判断，也不应继续提升为原创笔记候选。

### 6.5 Step 5：生成原创笔记候选

当用户完成文献转述后，系统才能帮助生成原创笔记候选。

原创笔记候选建议采用结构骨架，而不是完整正文，至少包括：

1. 核心判断。
2. 为什么成立。
3. 它依赖哪些论文证据。
4. 边界/反例/反驳。
5. 可连接的已有原创笔记。

系统在这里做的事是：

1. 帮用户压缩。
2. 帮用户组织。
3. 帮用户发现缺口。

而不是：

1. 替用户直接定稿。

### 6.6 Step 6：进入关联与洞见阶段

原创笔记形成后，才真正进入研思录原有优势区：

1. 支持关系建议。
2. 冲突关系建议。
3. 相似观点聚类。
4. 研究空白提示。
5. 可进入主题索引或写作项目。

洞见不应直接从 NotebookLM 摘要跳出来，而应从多条原创笔记的结构关系中长出来。

## 7. 界面建议

不建议只做一个“导入弹窗”。

更合适的是做一个 `论文工作台`，例如四栏布局：

1. 左栏：论文来源与 PDF。
2. 中左：NotebookLM 辅助内容与候选卡片。
3. 中右：我的转述与判断输入区。
4. 右栏：原创笔记候选与关联建议。

这样用户会感受到自己是在“加工理解”，而不是在“搬运摘要”。

## 8. NotebookLM 专属产品规则

### 8.1 来源绑定

所有 NotebookLM 输入内容都应绑定到某个明确 `Source`。

不能让它以“无来源文本”的方式进入系统。

### 8.2 痕迹保留

每条由 NotebookLM 引入的文献笔记候选，应保留至少这些痕迹：

1. 来自哪个 notebook。
2. 来自哪类产物：summary / Q&A / study guide / note。
3. 原始文本是什么。
4. 用户自己的转述是什么。

这样用户后续才不会混淆：

1. 哪些是外部整理结果。
2. 哪些是自己的理解。
3. 哪些已经足够提升为原创判断。

### 8.3 原创性守卫加强

凡是来源于 NotebookLM 的文本，在进入原创笔记前应更严格检查：

1. 是否只是对摘要做轻微改写。
2. 是否没有用户自己的判断。
3. 是否缺少边界条件。
4. 是否没有落到用户自己的研究问题上。

## 9. 不该做成什么样

这条工作流不应被设计成：

1. “导入 NotebookLM 内容成功”就算完成。
2. “NotebookLM 帮你写原创笔记”。
3. “把摘要收藏得更漂亮”。
4. “用户不需要做判断，只需要点接受”。

如果变成这样，产品就会退化成摘要搬运工具，而不是思想形成工具。

## 10. 成功标准

这条工作流真正成功，不应看这些指标：

1. 导入了多少条 NotebookLM 内容。
2. 收集了多少条摘要。
3. 粘贴了多少段文本。

更应该看这些指标：

1. 有多少 NotebookLM 候选被转成了用户自己的文献转述。
2. 有多少文献笔记进一步形成了原创笔记。
3. 有多少原创笔记进入了索引、关联或写作项目。
4. 用户是否能明确分清“外部整理结果”和“我的判断”。

## 11. 分阶段落地建议

### Phase 1

1. 论文 Source。
2. NotebookLM 文本粘贴导入。
3. 文献笔记候选拆分。
4. 转述表单。
5. 原创笔记候选生成。

### Phase 2

1. 自动关联建议。
2. 支持/冲突提示。
3. 跨论文比较。
4. 主题聚合与研究空白提示。

### Phase 3

1. Public Notebook 链接导入。
2. 更强的论文项目视图。
3. 从多篇论文生成研究问题地图。

## 12. 一句话结论

NotebookLM 在研思录里最合适的角色，不是替用户生成原创笔记，而是把论文阅读变成更容易被转述、被立论、被连接的中间层。
