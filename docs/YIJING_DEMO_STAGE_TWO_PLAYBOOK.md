# 易经官网 Demo 第二阶段讲述 Playbook

> 2026-05-16 叙事调整：本 playbook 保留为《易经》第二案例的讲述资料，不再作为官网主叙事或唯一 demo 入口。官网主站应展示研思录完整产品能力；新的 demo 规划见 `docs/OFFICIAL_SITE_AND_DEMO_REPLAN_2026-05-16.md`。

更新时间：2026-05-16

## 目标

第二阶段把易经 Demo 从“可以打开”推进到“可以讲清楚”。官网访客应该能在首页看到真实 Demo 入口，演示者应该能用 3 分钟说明研思录的核心价值：资料不是终点，显式关系、关联理由和写作方案才让笔记网络变成可追溯的判断。

## 3 分钟演示脚本

### 0:00 - 0:25 开场：不是资料库，而是判断网络

打开 `/demo`，先指出这套样例包含 `55` 条永久笔记、`85` 条显式关系、`2` 个写作方案。

讲法：

> 这个 Demo 不是展示《易经》资料库有多大，而是展示研思录如何让一组笔记变成可以继续思考和写作的网络。重点不是节点数量，而是每条关系都要说明“为什么要连”。

### 0:25 - 0:55 进入工作台：自动加载演示数据

点击“打开可交互 Demo”，进入 `/prototype?demo=yijing-rich`。等待状态提示出现 `已导入易经官网演示`，并确认自动切到关系图谱模块。

讲法：

> 官网 Demo 会自动导入样例数据，不需要用户先理解导入流程。这里看到的是永久笔记盒里的关系图谱，节点是判断，边是明确的思考关系。

### 0:55 - 1:35 第一层价值：关系类型不是标签装饰

打开关系类型筛选，展示 `supports`、`qualifies`、`counterexample_to`、`unexpected_connection`、`appears_in_draft` 等类型。

讲法：

> 普通笔记工具会告诉你两条笔记“相关”。研思录要追问它们到底怎么相关：是支持、限定、反例，还是只是一个值得继续追问的意外连接？关系类型让网络不是一团线，而是一张论证地图。

### 1:35 - 2:20 第二层价值：每条关系都有理由

选一条精选关系，强调 `rationale` 和 `insightQuestion`。

讲法：

> 一条线如果没有理由，就只是视觉连接。这里每条关系都保留了连接理由和后续问题：为什么这两条笔记值得一起看？这个连接还能推出什么新问题？这让图谱可以被审查，而不是只能被展示。

### 2:20 - 2:50 第三层价值：图谱进入写作

展示 `appears_in_draft` 或写作方案入口，说明写作篮、写作方案和脚手架的作用。

讲法：

> 进入写作方案不代表文章已经完成，它只表示这条判断被放进了论证现场。研思录保留来源、关系和段落位置，帮助写作者从已有判断出发，而不是重新面对空白页。

### 2:50 - 3:00 收束

讲法：

> 所以这个 Demo 展示的是一个完整动作：材料变成判断，判断彼此建立关系，关系再进入写作。研思录的核心不是替你生成答案，而是帮你保留形成判断的过程。

## 精选关系讲解素材

### 1. unexpected_connection：意外连接要产生追问

- Relation ID: `lnk_YJ-E12_YJ-C05_unexpected_connection`
- From: `YJ-E12 意外相关要留下追问`
- To: `YJ-C05 经典的含混是复用空间`
- Type: `unexpected_connection`
- Rationale: 经典含混与笔记网络的意外连接都依赖开放性，但意外相关必须提出可追问的问题。
- Insight question: 开放性什么时候产生洞见，什么时候变成任意联想？

讲解重点：

> 这条关系说明“意外相关”不是为了显得聪明。它必须留下一个可追问的问题，否则只是噪音。研思录把这种低成熟但有潜力的连接保留下来，同时提醒用户下一步要继续判断。

### 2. counterexample_to：反例让边界可见

- Relation ID: `lnk_YJ-E13_YJ-E06_counterexample_to`
- From: `YJ-E13 反例让边界可见`
- To: `YJ-E06 丰富笔记不是堆材料`
- Type: `counterexample_to`
- Rationale: 只堆支持材料的笔记是丰富笔记的反例，因为它缺少边界和反对意见。
- Insight question: 一条笔记需要多少反例才足以显示边界？

讲解重点：

> 这条关系适合展示研思录如何防止笔记网络变成“只支持自己”的材料堆。反例不是破坏观点，而是让观点知道自己在哪些条件下不成立。

### 3. appears_in_draft：进入写作不等于完成论证

- Relation ID: `lnk_YJ-E14_YJ-E05_appears_in_draft`
- From: `YJ-E14 进入写作不等于完成论证`
- To: `YJ-E05 写作方案要显示来源追溯`
- Type: `appears_in_draft`
- Rationale: 进入写作方案后，笔记仍需通过来源追溯说明它在段落中的论证位置。
- Insight question: 写作篮里哪些材料只是待用，哪些已经承担论证？

讲解重点：

> 这条关系适合作为 Demo 的收束。研思录不会把“放进写作篮”伪装成“已经写好”。它记录材料被征用，同时继续追问这条材料在段落里承担什么功能。

## 官网首页区块讲述方式

首页新增的 Demo 区块承担两个任务：

- 让访客知道官网里有一个真实可交互案例，而不是只有产品说明。
- 把 Demo 的价值提前压缩成三句话：关系类型、关联理由、写作准备。

推荐顺序：

1. 先看首页 Demo 区块，理解这不是普通截图。
2. 再进入 `/demo`，看到完整 Demo 页面。
3. 最后打开 `/prototype?demo=yijing-rich`，进入真实工作台。

## 正式演示环境风险

当前关系图谱模块默认展示永久笔记盒及其子目录。如果在已有开发 vault 中反复导入样例，图谱计数可能混入其他永久笔记或历史样例。

正式官网演示建议：

- 使用独立 `demo vault`，只包含官网演示数据。
- 每次录屏或销售演示前重建 demo vault，避免历史数据影响计数。
- 后续可增加 `demo-only scope`，让 `/prototype?demo=yijing-rich` 自动把图谱范围锁定到 `dir_yijing_rich_acceptance_original`。
- 如果使用共享环境，页面文案应避免承诺图谱总数永远等于 55/85，而改说“演示数据包含 55 条永久笔记和 85 条关系”。
