# 研思录 MVP 执行计划（4周）v1.0

## 1. 目标

在 4 周内交付可演示的桌面端 MVP，跑通主链路：

`资料输入 -> 书摘笔记 -> 永久笔记 -> 索引与连接 -> 当前目录图谱 -> 写作提示词与提纲`

并满足以下硬约束：

1. Markdown 为主内容事实源。
2. `[[标题]]` 显示，SQLite 内部绑定 `note_id`。
3. 相似度 `>= 80%` 时阻止保存永久笔记文件。
4. 写作入口仅允许永久笔记目录下的永久笔记。
5. 导入仅做 Markdown / Obsidian 基础导入（MVP）。

---

## 2. 技术与代码布局

技术栈建议：`Tauri + React + TypeScript + SQLite + CodeMirror 6`

代码布局（已确认）：

- `apps/desktop`：Tauri 壳与本地能力桥接
- `apps/web`：原型/界面层（可复用组件）
- `apps/api`：本地 BFF（如需）
- `packages/domain`：领域模型与规则
- `packages/markdown-engine`：Markdown 读写/解析
- `packages/search-engine`：基础检索
- `packages/graph-engine`：当前目录图
- `packages/originality-guard`：相似度检测与阻断
- `schemas/*.json`：结构化对象约束
- `docs/*.md`：规范与验收

---

## 3. 并行开发流（建议 4 条）

1. Core 流：目录、Markdown、SQLite 基础能力
2. Editor 流：统一 Markdown 编辑器与 `[[`/`#` 交互
3. Workflow 流：随笔/书摘/永久笔记流程 + 写作中心
4. Import 流：Markdown/Obsidian 导入（预览->确认）

规则：

1. 先冻结 docs 与 schema，再开大规模代码改动。
2. 每条流每天至少合入一个小切片，避免超大 PR。
3. 所有接口先有 contract 再写实现。

---

## 4. 4周里程碑

## Week 1：基础骨架与数据闭环

目标：先把“能存、能读、能关联”打通。

交付：

1. 目录模型落地（默认三目录 + 子目录管理 + 本地路径绑定）。
2. Markdown 文件 CRUD（标题=第一行）。
3. SQLite 表落地（notes/directories/links/tags/index/writing）。
4. `[[标题]]` 解析并绑定 `note_id`。
5. `#标签` 解析入库。
6. 原型 UI：左导航 + 文件树 + 多标签编辑区。

验收出口：

1. 新建目录和笔记后，重启应用可恢复。
2. 编辑器插入 `[[` 可检索并绑定目标笔记。
3. 标签点击可列出当前目录内同标签笔记。

## Week 2：核心工作流

目标：完成随笔、书摘、永久笔记三类流程。

交付：

1. 随笔记录（文本/语音占位）与 7 天隐藏策略。
2. 随笔转书摘、随笔转永久笔记。
3. 书摘笔记字段集与保存校验。
4. 从书摘新增原创，且书摘保持原状态。
5. 原创性守卫接入保存流程（`>=80%` 阻断）。

验收出口：

1. 流程 A/B/C（验收文档）本地跑通。
2. 阻断时文件不写入，提示原因可理解。

## Week 3：结构与写作

目标：把“可写结构”交付出来。

交付：

1. 索引笔记（特殊永久笔记）与排序。
2. 每净增 15 条永久笔记触发索引整理提醒。
3. 当前目录图谱（节点、边、过滤）。
4. 写作中心：选择标签/永久笔记 -> 生成提示词 -> 生成提纲。

验收出口：

1. 仅永久笔记目录可进入写作流程。
2. 图谱可回跳笔记详情。
3. 写作结果可追溯到输入笔记集。

## Week 4：导入、稳定性与发布准备

目标：做可演示版本而不是半成品。

交付：

1. Markdown/Obsidian 基础导入（预览->确认->写入）。
2. 导入记录与冲突提示。
3. 回归测试脚本（A0~L 关键流程）。
4. 性能与稳定性压测（含 10,000 永久笔记模拟数据）。
5. 打包桌面预发布版本（Win/macOS/Linux）。

验收出口：

1. 无 P0 缺陷。
2. 完整演示从空目录到提纲输出可跑通。

---

## 5. 日常协作机制

1. 每日 15 分钟站会：昨天完成、今天目标、阻塞项。
2. 每晚合并前跑一次最小回归：
- 新建目录
- 新建书摘
- 书摘转永久笔记
- `[[` 关联
- 标签检索
- 原创性阻断
3. 每两天一次文档校对：`PRODUCT_SPEC`、`ARCH`、`ACCEPTANCE` 与实现一致。

---

## 6. 风险与应对

1. 风险：范围膨胀。
应对：严格执行 MVP 边界，推迟非 Markdown 导入器。
2. 风险：UI 持续返工。
应对：先锁定信息架构与交互骨架，再做视觉细化。
3. 风险：关系解析复杂导致不稳定。
应对：先支持 `[[标题]]` 与 `#标签` 最小规则，不引入复杂语法。
4. 风险：性能退化。
应对：图谱默认局部、检索分页、缓存可重建。

---
