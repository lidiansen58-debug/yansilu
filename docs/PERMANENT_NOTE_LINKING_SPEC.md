# 研思录永久笔记关联系统 Spec

## 1. 文档目标

本文档定义研思录永久笔记之间的关联系统。

它不是一个普通链接工具，而是一个：

`思想生长反馈系统`

系统目标不是让用户多做一步整理，而是让用户在建立关系时感到：

1. 这条思想接入了网络。
2. 一个以前没看见的关系出现了。
3. 某个主题开始形成结构。
4. 一组笔记正在靠近写作。

---

## 2. 核心判断

永久笔记的价值不只在单条判断是否清楚，还在它与其他判断之间能否形成结构。

因此，关联系统必须回答五个问题：

1. 这条原创判断与哪些判断相关？
2. 它们是什么关系？
3. 为什么是这种关系？
4. 这条关系暴露了什么新问题？
5. 这条关系能否进入主题索引、图谱或写作结构？

一条高质量关系的最小表达应是：

`A 与 B 的关系是 X，因为 Y；这提示了 Z。`

其中：

1. `X` 是关系类型。
2. `Y` 是连接理由。
3. `Z` 是洞见问题或下一步思考入口。

---

## 3. 体验原则

## 3.1 从当前节点看网络

当用户查看一条永久笔记时，界面不应只展示这条笔记本身。

它应相当于：

`从当前节点的位置，向整个思想网络望去。`

因此，永久笔记详情页必须提供一个当前节点视角：

1. 向上看：哪些笔记是它的前提、来源、支持？
2. 向下看：哪些笔记被它推进、限定、应用？
3. 向旁边看：哪些笔记与它互补、对比、同问异答？
4. 向张力看：哪些笔记与它冲突、构成反例或边界？
5. 向写作看：它能进入哪个主题、写作篮或脚手架？

## 3.2 不允许未解释的孤岛

永久笔记不应长期处于没有任何位置说明的孤岛状态。

但系统不应强迫用户为了过关硬连一条低质量关系。

正确规则是：

`系统强制用户处理孤岛状态，但不强制制造连接。`

当一条永久笔记没有确认关系、没有进入主题索引、也没有写作归属时，系统必须要求用户完成四选一：

1. 建立至少一条有类型、有理由的连接。
2. 加入一个主题索引。
3. 标记为思想种子，并写一句未来可能连接方向。
4. 标记为暂不接入，并写一句原因。

这样，强制的是：

`交代它在思想网络中的状态。`

而不是：

`为了消除孤岛而随便连边。`

## 3.3 连接不是数量成就

系统不奖励连接数量。

系统奖励：

1. 有理由的连接。
2. 产生张力的连接。
3. 跨主题桥接。
4. 能进入写作结构的连接。
5. 让主题更清楚的连接。

---

## 4. 关系方法

## 4.1 三步建立关系

用户建立关系时，系统应引导三步：

1. 选择相关笔记。
2. 选择关系类型。
3. 写一句连接理由。

在高价值关系中，系统还应给出第四步：

4. 写下这条关系提出的新问题。

## 4.2 连接理由模板

空白输入框会提高阻力。

系统应按关系类型提供填空式提示：

| 关系 | 连接理由提示 |
|---|---|
| 支持它 | 这条笔记支持另一条，是因为... |
| 补充它 | 这条笔记补充另一条，是因为它提供了... |
| 对比它 | 这两条笔记讨论相近问题，但差异在于... |
| 反驳它 | 这两条笔记产生冲突，是因为... |
| 推进它 | 这条笔记把另一条推进到... |
| 限定它 | 这条笔记缩小或修正另一条的适用范围，因为... |
| 提供反例 | 这条笔记显示另一条在这个情况下不成立... |
| 意外相关 | 它们看似不相关，但都在讨论... |
| 连接主题 | 这条笔记把两个主题连接起来，因为... |

## 4.3 洞见问题模板

系统不应替用户生成结论。

系统可以生成问题：

| 关系场景 | 洞见问题模板 |
|---|---|
| 冲突 | 如果这两条都部分成立，真正的分歧条件是什么？ |
| 反例 | 这条反例说明原判断需要加上什么边界？ |
| 限定 | 原判断在哪些条件下成立，在哪些条件下不成立？ |
| 意外连接 | 它们共同指向了哪个更深的问题？ |
| 跨主题桥接 | 这条连接是否能发展成一个新主题入口？ |
| 递进链 | 这组三步关系能否形成一段论证？ |

---

## 5. 关系类型

## 5.1 推荐内部枚举

关联系统建议支持以下内部关系类型：

| 内部值 | UI 文案 | 说明 |
|---|---|---|
| `supports` | 支持它 | A 为 B 提供理由、证据或论证支撑 |
| `complements` | 补充它 | A 补充 B 的另一个角度 |
| `contrasts` | 对比它 | A 和 B 讨论相近问题，但角度不同 |
| `contradicts` | 反驳它 | A 与 B 存在观点张力或不兼容 |
| `extends` | 推进它 | A 在 B 基础上展开、应用或推进 |
| `precedes` | 作为前提 | A 是理解 B 的前置判断 |
| `follows` | 作为后续 | A 是 B 后续自然推出的问题 |
| `qualifies` | 限定它 | A 修正、缩小或限定 B 的适用范围 |
| `example_of` | 作为例子 | A 是 B 的具体例子 |
| `counterexample_to` | 提供反例 | A 是 B 的反例、边界或挑战 |
| `same_topic` | 同属主题 | A 与 B 属于同一主题，但关系尚不明确 |
| `unexpected_connection` | 意外相关 | A 与 B 看似远，但可能产生新问题 |
| `bridges` | 连接主题 | A 桥接两个主题或两个判断群 |
| `restates` | 重述它 | A 用另一种表述或层级重新说明 B |
| `reframes` | 改变问题 | A 改变了理解 B 的问题框架 |
| `appears_in_draft` | 进入写作 | A 被写作项目或脚手架引用 |

## 5.2 与现有类型的兼容

当前系统已有：

1. `supports`
2. `contradicts`
3. `extends`
4. `precedes`
5. `follows`
6. `belongs_to_topic`
7. `associated_with`
8. `appears_in_draft`
9. `free_link`

迁移或扩展时建议：

1. `associated_with` 可以作为 `same_topic` 或 `unexpected_connection` 的低成熟度前身。
2. `free_link` 应逐步被更明确的关系类型替代，但仍可保留为用户直觉池。
3. `belongs_to_topic` 更像索引归属关系，不应混同于两条永久笔记之间的语义关系。
4. `counterexample_to` 比泛用的 `contradicts` 更适合表达边界和反例。
5. `restates` 只用于确认同一判断的更清晰表达，不应替代 `supports` 或 `extends`。

## 6. Link 数据结构建议

## 6.1 最小字段

```ts
interface Link {
  id: string;
  from_note_id: string;
  to_note_id: string;
  relation_type: string;
  rationale: string;
  insight_question?: string;
  created_by: "user" | "ai_suggestion" | "import";
  confidence?: number;
  status: "suggested" | "draft" | "confirmed" | "dismissed" | "archived";
  created_at: string;
  updated_at: string;
}
```

## 6.2 字段规则

1. `relation_type` 必填。
2. `rationale` 对用户确认的显式关系必填。
3. `insight_question` 对冲突、反例、意外连接、跨主题桥接建议强提示。
4. `created_by = ai_suggestion` 的关系默认只能是 `suggested`。
5. AI 不得自动写入 `confirmed`。
6. `status = dismissed` 应保留，以避免 AI 反复推荐同一低价值关系。

---

## 7. 节点状态

每条永久笔记应有一个派生的网络状态。

| 状态 | 含义 | 触发条件示例 |
|---|---|---|
| `unplaced` | 未安置 | 没有关系、索引、种子说明或暂不接入理由 |
| `seed` | 思想种子 | 用户确认暂时孤立，但写下未来连接方向 |
| `connected` | 已连接 | 至少一条确认关系 |
| `topic_node` | 主题节点 | 已进入至少一个主题索引 |
| `bridge_node` | 桥接节点 | 连接两个以上主题或判断簇 |
| `core_node` | 核心节点 | 被多条笔记支持、引用或推进 |
| `writing_node` | 写作节点 | 已进入写作篮、写作项目或脚手架 |
| `parked` | 暂不接入 | 用户说明暂不接入原因 |

状态变化时应给出克制正反馈：

| 变化 | 反馈文案 |
|---|---|
| `unplaced -> connected` | 这条思想接入了网络。 |
| `unplaced -> seed` | 这条思想先作为种子保留，等待未来连接。 |
| `connected -> topic_node` | 这条思想进入了一个主题结构。 |
| `topic_node -> bridge_node` | 这条思想开始连接两个主题。 |
| `topic_node -> writing_node` | 这条思想已经开始参与写作。 |

---

## 8. 孤岛处理规则

## 8.1 触发时机

当用户查看一条 `unplaced` 永久笔记时，系统显示孤岛处理面板。

标题：

`这条思想还没有位置`

说明：

`为它建立一个关系，或说明它为什么暂时独立存在。`

操作：

1. `建立连接`
2. `加入主题索引`
3. `标记为思想种子`
4. `暂不接入`

## 8.2 强制级别

建议分三档：

| 档位 | 行为 |
|---|---|
| 轻提醒 | 新保存后提示，但允许关闭 |
| 强提醒 | 多次查看仍未处理时，详情页持续显示 |
| 必须处理 | 用户要把该笔记放入写作篮、主题成熟度统计或图谱核心视图前，必须处理 |

## 8.3 不允许的行为

1. 不允许系统自动为孤岛创建确认连接。
2. 不允许用户一键批量把孤岛标记为已处理。
3. 不允许无理由地把孤岛加入主题索引。
4. 不允许图谱把未解释孤岛伪装成成熟结构。

---

## 9. 保存后的关联时刻

用户保存永久笔记后，不只显示保存成功。

系统应出现轻量面板：

`让这条思想接入你的网络`

面板展示 3 到 5 条候选关联：

1. 候选笔记标题。
2. 简短摘要。
3. 推荐关系类型。
4. 推荐理由。
5. 推荐洞见问题。
6. 操作按钮：建立连接、换一种关系、忽略、稍后处理。

示例：

```text
候选笔记：AI 不应代写永久笔记
推荐关系：支持它
推荐理由：两条笔记都在说明思想主权不能外包给 AI。
可能问题：AI 辅助与思想主权之间的边界在哪里？
```

确认后反馈：

`连接已建立。你的思想网络新增了一条支持关系。`

---

## 10. 当前节点关系面板

永久笔记详情页右侧应提供关系面板。

## 10.1 概览

显示：

1. 网络状态。
2. 已连接数量。
3. 支持、补充、冲突、反例、桥接等关系数量。
4. 所属主题。
5. 是否为桥接节点、核心节点、写作节点。

## 10.2 当前节点视角

关系面板应分区显示：

1. 前提与来源。
2. 支持与证据。
3. 补充与对比。
4. 冲突与反例。
5. 推进与后续。
6. 主题与写作入口。

## 10.3 关系详情

点击关系时显示：

1. 两张笔记。
2. 关系类型。
3. 连接理由。
4. 洞见问题。
5. 是否用于主题索引或写作项目。

---

## 11. 洞见候选卡

洞见候选不是 AI 替用户写结论。

它是系统发现高价值关系后，给用户留下的思考入口。

## 11.1 触发条件

以下关系应触发洞见候选：

1. `contradicts`
2. `counterexample_to`
3. `qualifies`
4. `unexpected_connection`
5. `bridges`
6. `reframes`
7. 三条以上 `precedes/follows/extends` 构成递进链
8. 两个长期无关主题之间第一次出现连接

## 11.2 卡片内容

洞见候选卡包含：

1. 触发来源。
2. 关系类型。
3. 用户写下的连接理由。
4. AI 或系统生成的问题。
5. 用户输入框：`我的新想法是...`
6. 操作：写成新永久笔记、加入主题索引、加入写作篮、稍后处理、忽略。

## 11.3 文案原则

不要写：

`你的洞见是...`

应写：

`这条连接提出了一个问题...`

或者：

`这里可能出现了一条新的思考线索。`

---

## 12. 主题成熟度

主题索引应有成熟度阶段。

| 阶段 | 含义 |
|---|---|
| `seed` | 只有 1 到 2 条笔记，主题刚出现 |
| `sprout` | 有 3 到 5 条笔记，开始有连接 |
| `branch` | 有多个子方向和关系类型 |
| `grove` | 结构较完整，有支持、冲突、来源与边界 |
| `writable` | 已足以进入写作篮或生成脚手架 |

计算维度：

1. 永久笔记数量。
2. 连接数量。
3. 关系类型丰富度。
4. 是否有支持关系。
5. 是否有冲突、反例或边界。
6. 是否有来源引用。
7. 是否有核心观点。
8. 是否已经进入写作项目。

反馈示例：

`AI 与写作主题已从 sprout 成长为 branch。新增 3 条支持关系和 1 条反方观点。`

---

## 13. 写作准备度

当主题成熟后，系统应提示写作准备度。

显示：

1. 核心观点。
2. 支持材料。
3. 反方观点。
4. 边界或反例。
5. 来源引用。
6. 缺口。
7. 推荐写作方向。

示例：

```text
写作准备度：72%

已具备：
- 核心观点 3 条
- 支持材料 5 条
- 反方观点 1 条
- 来源引用 4 条

待补充：
- 真实案例不足
- 过渡笔记不足

可写方向：
1. 为什么笔记不是收藏，而是思想生长系统
2. AI 时代为什么更需要自己写永久笔记
```

---

## 14. 成就反馈

成就反馈应克制，避免游戏化过度。

## 14.1 即时反馈

用于单条关系或单条笔记：

1. 这条思想接入了网络。
2. 它找到了第一位思想邻居。
3. 这条连接提出了一个新问题。

## 14.2 结构反馈

用于主题或局部图谱：

1. 这个主题开始形成结构。
2. 这里出现了一条冲突线。
3. 这条笔记正在连接两个主题。

## 14.3 成果反馈

用于写作转化：

1. 这个主题已经可以进入写作篮。
2. 你已经不是在积累资料，而是在准备一篇文章。
3. 基于当前关联，可以生成一个初稿脚手架。

## 14.4 长期成长报告

每周报告建议展示：

1. 本周新增永久笔记。
2. 本周新增确认连接。
3. 本周孤岛减少数。
4. 本周新形成主题。
5. 本周发现冲突。
6. 本周洞见候选。
7. 本周可写主题。

---

## 15. AI 参与边界

AI 可以做：

1. 找候选关联。
2. 解释为什么相关。
3. 建议关系类型。
4. 发现孤岛、桥接缺口和主题缺口。
5. 生成洞见问题。
6. 生成写作准备度说明。

AI 不可以做：

1. 自动确认连接。
2. 自动把 suggested 关系改为 confirmed。
3. 无理由批量连边。
4. 自动写入用户永久笔记正文。
5. 把洞见问题改写成用户已经接受的结论。

---

## 16. MVP 优先级

## P0

1. 关系类型系统。
2. 连接理由记录。
3. 保存永久笔记后的关联时刻。
4. 当前节点关系面板。
5. 孤岛处理面板。
6. 主题索引基础关系统计。

## P1

1. 节点状态。
2. 主题成熟度。
3. 洞见候选卡。
4. 写作准备度。
5. AI 候选关联。

## P2

1. 跨主题桥接发现。
2. 冲突视图。
3. 关系质量评分。
4. 每周思想网络报告。
5. 长期思想网络成长曲线。

---

## 17. 验收标准

1. 用户能为两张永久笔记建立带类型和理由的连接。
2. 用户保存永久笔记后能看到候选关联。
3. 用户可以确认、改写、忽略或延后 AI 建议。
4. 未安置永久笔记在详情页会触发孤岛处理面板。
5. 用户可以不建立连接，但必须说明该笔记的网络状态。
6. 永久笔记详情页能从当前节点视角展示前提、支持、冲突、后续和写作入口。
7. 高价值关系能生成洞见候选问题，但不生成用户结论。
8. 主题索引能统计关系结构和成熟度。
9. 写作篮能使用连接关系生成带证据、张力和缺口的脚手架。

---

## 18. 禁止事项

1. 不允许 AI 自动确认连接。
2. 不允许无理由的大量自动连边。
3. 不允许为了消除孤岛而强迫用户乱连边。
4. 不允许图谱默认展示全量大图。
5. 不允许把连接系统退化成标签系统。
6. 不允许把连接数量作为主要成就指标。

---

## 19. 核心产品感觉

用户每建立一条高质量关系，都应该感觉：

`我刚刚让自己的思想网络长大了一点。`

这套系统最终要提供三种反馈：

1. 即时反馈：这条笔记接入网络了。
2. 结构反馈：这个主题开始成熟了。
3. 成果反馈：这个主题可以写了。

三者合起来，永久笔记关联才会从知识管理动作变成持续使用的思考体验。
