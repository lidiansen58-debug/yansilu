# Prototype App Modularization Map

本文档记录 `apps/web/src/prototype-app.js` 的当前结构、迁移边界和建议提交顺序。目标不是一次性拆完，而是把大文件变成装配层：初始化状态、连接模块、绑定全局事件，其余领域逻辑逐步外移。

## 当前职责地图

`prototype-app.js` 当前约 2.3 万行，主要职责集中在以下区域：

## 建议迁移顺序

1. **纯 helper 先行**
   - 模板/草稿 helper
   - 图谱关系 key、endpoint、状态判断
   - 图谱候选排序/合并/文案推导
   - 目标：无 DOM、无 API、无全局状态写入。

2. **低风险渲染模块**
   - 图谱孤立笔记关系浮层渲染
   - 图谱选择面板中的小块渲染
   - 写作中心状态条和写作篮摘要
   - 目标：输入数据 -> HTML 字符串，测试用快照或文本断言覆盖。

3. **单一功能控制器**
   - 图谱孤立笔记处理流程
   - 图谱关系候选保存桥接
   - 系统消息工作流桥接
   - 目标：通过显式依赖注入拿 API、状态读写和 UI 刷新函数。

4. **最后拆事件和启动**
   - 全局事件代理
   - 组件实例装配
   - Bootstrap
   - 目标：等领域模块稳定后，只保留应用装配逻辑。

## 近期 5 个提交边界

1. 固定 `prototype-app.js` 换行策略，清理状态噪声。
2. 建立本文档，明确大文件职责地图和迁移边界。
3. 抽出纯 helper，优先选图谱关系 key/endpoint/status 一类无副作用函数。
4. 抽出第一个低风险功能模块，优先选图谱关系候选/孤立笔记浮层的纯渲染或表单状态。
5. 抽出第二个低风险功能模块，优先选 graph follow-up 或系统消息桥接中的单一入口。

## 不应优先迁移的部分

- `bootstrap()` 和组件实例化。
- 全局事件代理的大段拆分。
- 同时读写 `state`、`graphState`、`writingState` 的跨模块函数。
- 需要变更用户流程的重构。

这些区域应该在 helper、渲染模块、功能控制器逐步稳定后再处理。

## 验证要求

每一批迁移至少满足：

- `node --check apps/web/src/prototype-app.js`
- 新模块对应的 `node --check`
- 相关单元测试
- 如果触达用户流程，补跑对应 browser e2e 分组或单条路径
