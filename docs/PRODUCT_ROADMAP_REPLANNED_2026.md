# 研思录产品发展 Roadmap（重规划版，2026-05-09）

## 1. 文档目标

本文档用于重新规划研思录未来发展的产品 roadmap。

它不再以“先做哪些功能、再做哪些功能”的工程任务视角为主，而改为以“产品能力发生了什么跃迁”为主。

因此，这份 roadmap 的核心问题不是：

1. 我们还能继续加什么？
2. 哪些功能更容易做？

而是：

1. 研思录下一步要先成为什么？
2. 哪些能力一旦做早了，会把产品带偏？
3. 哪些阶段完成后，产品的性质会真正改变？

---

## 2. 长期北极星

研思录的长期目标不是成为：

1. 更强的笔记软件
2. 更炫的图谱工具
3. 更快的 AI 写作工具

而是成为：

`写作与知识创作领域的 Codex`

它服务的不是“内容堆积”，而是“知识创作”。

它帮助用户完成的不是“记录更多”，而是：

`来源 -> 转述 -> 判断 -> 关系 -> 主题 -> 结构 -> 作品`

同时必须长期守住两条根本原则：

1. `原创判断归用户`
2. `AI 帮助思考，不代替思想`

---

## 3. 路线图的重排原则

未来 roadmap 必须遵循以下顺序：

1. 先建立可信的本地知识工作流
2. 再建立思想提纯层
3. 再建立主题级判断与冲突组织
4. 再引入 AI 作为知识创作协同层
5. 最后才进入“知识创作界 Codex”的成熟形态

这意味着：

1. 不能先做重型 AI 自动生成
2. 不能先做全局图谱炫技
3. 不能先做大而全连接器生态
4. 不能先做团队协作平台化

因为这些都会让产品过早偏向：

1. 仓库化
2. 平台化
3. 代写化
4. 展示化

而不是思维与知识创作工具。

---

## 4. 当前阶段判断

截至 `2026-05-09`，研思录大致处于：

`Phase 0.8：可信 MVP 已基本成形，但思想提纯层尚未真正产品化`

当前已具备的基础：

1. 本地工作区与目录树
2. 三类笔记基础流转
3. 导入预览、确认、历史、回滚
4. 原创性守卫
5. 关联、标签、图谱、写作项目与脚手架
6. Web 原型 + API + 自动化测试主链路

当前最缺的不是更多“记录”能力，而是：

1. 判断压缩
2. 主题级组织
3. 边界/反方/未解之处
4. AI 候选态认知辅助

因此 roadmap 应从这里重新起跳。

---

## 5. 重规划后的五阶段路线图

## Phase 0：可信知识工作流

### 阶段定义

让用户相信：

`这套系统真的能安全承载我的本地思考与写作资产。`

### 产品关键词

1. 本地优先
2. 文件可读
3. 结构可恢复
4. 导入可回滚
5. 写作可追溯

### 阶段目标

1. 跑通本地工作区与三类笔记主流程
2. 跑通导入、回滚、导出
3. 跑通永久笔记到脚手架的闭环
4. 让桌面壳能力稳定可用

### 成功标准

1. 用户敢把真实资料和真实笔记放进来
2. 用户知道文件始终归自己所有
3. 产品不会静默损坏本地内容
4. 原创性守卫不会被轻易绕过

### 阶段结论

这不是终点，而是后续所有提纯和 AI 协同的可信底座。

---

## Phase 1：思想提纯器 V1.1

### 阶段定义

让研思录第一次从“可记录、可组织”变成“会逼用户想清楚”。

### 核心命题

`没有判断压缩，就没有真正的知识形成。`

### 重点能力

1. `PermanentNote.thesis`
2. `PermanentNote.three_line_summary`
3. `IndexCard.thesis`
4. `IndexCard.three_line_summary`
5. `IndexCard.central_question`
6. `WritingProject.intent`
7. `desired_reader_takeaway`

### 用户价值

1. 帮助用户把“这条笔记想说什么”压成一句话
2. 帮助用户把“为什么重要”说明白
3. 帮助用户把“这个主题的核心问题是什么”提出来
4. 帮助用户在写作前先澄清表达意图

### 本阶段不追求

1. 更强连接器
2. 更大图谱
3. 更重自动化
4. AI 直接成文

### 阶段成功标准

1. 用户愿意高频补全 `thesis`
2. 用户愿意围绕主题填写 `central_question`
3. 写作项目明显更多从已有判断出发，而不是从空白页开始

### 阶段意义

这是产品从“知识整理工具”向“判断形成工具”跃迁的第一阶段。

---

## Phase 2：主题与张力操作系统

### 阶段定义

让产品不只支持“单条判断”，还支持“主题级思考”。

### 核心命题

`真正的知识创作，不只来自观点积累，更来自对主题张力、冲突和边界的组织。`

### 重点能力

1. 索引卡正式升级为主题思考对象
2. 主题图、写作图、张力图更清晰分工
3. 关系类型化：
   - 支持
   - 冲突
   - 前提
   - 延伸
   - 反例
   - 待验证
4. 显式记录：
   - 边界
   - 反方
   - 未解问题
   - 桥接缺口

### 用户价值

1. 不再只是“很多观点”
2. 开始形成“一个主题到底在争论什么”
3. 能看见概念错位、断裂、重复和冲突
4. 能让写作从主题判断和问题张力出发

### 阶段成功标准

1. 用户开始围绕主题组织永久笔记
2. 图谱不再只是“有连接”，而是“有结构意义”
3. 写作篮的进入方式更多来自主题索引，而不是手工拼装

### 阶段意义

这是产品从“判断卡片集合”跃迁为“主题思考系统”的阶段。

---

## Phase 3：AI 认知协同层

### 阶段定义

让 AI 真正进入工作流，但只扮演认知协同角色，而不是思想代写角色。

### 核心命题

`AI 的价值不是替用户拥有观点，而是帮助用户更快看见结构、盲点、冲突与洞见。`

### 重点能力

基于 AI Agent SDK，逐步引入以下候选态能力：

1. 笔记阅读助手
2. 主题关系分析员
3. 中心问题候选生成
4. 一句话/三句话压缩候选
5. 反方与边界提醒
6. 冲突/重复/桥接缺口分析
7. 写作结构顾问

### 关键约束

1. AI 只对选定范围工作：
   - 当前笔记
   - 当前主题
   - 当前写作项目
   - 当前来源集合
2. 所有输出默认是 suggestion
3. 所有核心建议必须附 `rationale`
4. 永久笔记正文不能由 AI 自动写入并落库

### 用户价值

1. 更快看清已有笔记里真正的模式
2. 更快发现自己忽略的张力与边界
3. 更快从一堆判断进入结构化写作准备

### 阶段成功标准

1. AI 建议被认为“有启发”，而不是“很会写但没帮助”
2. AI 没有抢走用户的主体性
3. 用户开始把 AI 当“思维陪练”，不是代写员

### 阶段意义

这是产品从“纯工具”跃迁为“认知协同系统”的阶段。

---

## Phase 4：知识创作工作台

### 阶段定义

让研思录不只停在“思考和脚手架”，而开始支持稳定的知识作品生产。

### 核心命题

`知识创作不是一次成文，而是围绕判断、结构、版本、证据和修正不断生长。`

### 重点能力

1. 写作项目强化：
   - 意图
   - 受众
   - 读者带走判断
   - 版本关系
2. 草稿笔记与脚手架绑定
3. 章节级证据映射更清晰
4. 支持作品级结构：
   - 长文
   - 系列文章
   - 课程草案
   - 研究专题
5. 记录判断演化与修订轨迹

### 用户价值

1. 从“有很多想法”推进到“产出真正作品”
2. 不丢失来源追溯
3. 不被 AI 抹平个人立场
4. 能回看自己判断如何生长和变化

### 阶段成功标准

1. 用户能稳定从既有判断结构产出长作品
2. 写作链路越来越少依赖空白 prompt
3. 用户感受到产品在帮助自己“持续创作”，而不是一次性生成

### 阶段意义

这是产品从“思维工具”跃迁为“知识创作工作台”的阶段。

---

## Phase 5：知识创作界的 Codex

### 阶段定义

让研思录成为围绕知识创作全过程的长期系统。

### 核心命题

`生成对象不是代码，而是内容背后的判断、结构、关系与作品体系。`

### 这一阶段可能出现的能力

1. 多 Agent 协同分析主题与项目
2. 基于用户知识空间的深度研究辅助
3. 作品级知识资产管理
4. 更成熟的创作路径建议
5. 跨主题、跨项目的长期思维演化视图

### 但必须始终守住

1. 原创判断归用户
2. AI 输出不是最终思想所有权
3. 数据主权归用户
4. 可追溯、可解释、可回退

### 阶段意义

这是“知识创作界 Codex”愿景真正落地的成熟形态。

---

## 6. 贯穿全路线的三条长期主线

除了阶段推进，还应始终并行推进三条横向主线：

### 6.1 数据主权主线

始终坚持：

1. 本地优先
2. Markdown 可读
3. 结构可导出
4. AI 结果可追溯

### 6.2 产品哲学主线

始终坚持：

1. 不奖励收集，奖励提炼
2. 不奖励流畅，奖励澄清
3. 不奖励连接数量，奖励连接理由
4. 不奖励生成，奖励承担

### 6.3 AI 边界主线

始终坚持：

1. AI 只做认知辅助
2. AI 不自动拥有观点
3. AI 不静默改写核心知识对象
4. AI 建议必须是候选态

---

## 7. 明确不按 roadmap 优先推进的事项

以下事项不应成为未来 1 到 2 个阶段的优先核心：

1. 团队协作平台化
2. 大而全插件市场
3. 默认全局图谱浏览器
4. 强依赖云端的知识托管
5. AI 一键生成永久笔记正文
6. AI 一键生成完整终稿
7. 以导入量、收藏量为中心的增长策略

原因很简单：

这些方向虽然可能带来“看起来更大”的产品形态，但会削弱研思录最独特的价值。

---

## 8. 当前建议的真实推进顺序

如果从现在开始重新排顺序，我建议按以下次序推进：

1. `收口 Phase 0`
2. `进入 Phase 1`
3. `再推进 Phase 2`
4. `谨慎开启 Phase 3`
5. `最后再进入 Phase 4 与 Phase 5`

换成更直白的话就是：

1. 先把 MVP 做成真正可信的底座
2. 再把思想提纯做成产品主链路
3. 再把主题和张力组织做出来
4. 再把 AI 接进来做认知协同
5. 最后把它推进为真正的知识创作系统

---

## 9. 一句话版 roadmap

`先做可信的本地知识工作流，再做思想提纯器，再做主题与张力操作系统，再做 AI 认知协同层，最后成为知识创作界的 Codex。`
