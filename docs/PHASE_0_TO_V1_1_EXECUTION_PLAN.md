# Phase 0 收口到 V1.1 观点提纯执行计划

> Status: working plan
> Date: 2026-05-16
> Branch: `codex/wt-ai-harness`

## 1. 推进判断

当前推进顺序保持为：

```text
先收口 Phase 0 可信 MVP -> 再推进 Phase 1 / V1.1 观点提纯
```

原因：

1. Phase 0 是后续提纯、关系、图谱和 AI 协同的可信底座。
2. V1.1 的核心不是继续加入口，而是把 `thesis`、`three_line_summary`、`central_question`、`intent` 推进主链路。
3. AI 能力已经具备 review-only 的底座，但产品上仍应先服务提纯动作，而不是直接扩大为重型 Agent。

## 2. 包 A：Phase 0 MVP 收口

目标：

```text
让用户相信这套系统真的能安全承载本地思考与写作资产。
```

### 2.1 验收范围

必须覆盖：

1. 本地工作区初始化、切换与恢复。
2. 三类笔记主流程：Source、LiteratureNote、PermanentNote。
3. 导入预览、确认、历史、回滚。
4. 导出 Markdown。
5. 原创性守卫。
6. 永久笔记关联、标签、图谱基础查看。
7. 永久笔记进入写作篮、创建写作项目、生成脚手架。
8. 桌面壳关键能力：选择 vault、打开文件、外链打开。
9. 自动保存、dirty 状态、草稿恢复。
10. 文件路径、中文文件名、空格路径、资产链接。

### 2.2 真实验收流

用一条端到端用户流验收：

```text
来源资料 -> 文献笔记 -> 永久笔记 -> 关系/标签 -> 图谱查看 -> 写作项目 -> 脚手架 -> 导出/回滚
```

验收时重点看：

1. 不丢数据。
2. 不静默改写用户正文。
3. 文件路径可恢复、可读、可迁移。
4. 导入失败、回滚失败、保存失败都有明确提示。
5. 原创性守卫不能被主流程轻易绕过。

### 2.3 收口规则

Phase 0 收口期间冻结非必要新功能，只修：

1. 数据损坏风险。
2. 保存、恢复、导入、回滚、导出不稳定。
3. 真实主流程阻断。
4. 用户无法理解的关键状态。
5. 自动化测试覆盖缺口。

不做：

1. 新的大型 AI 能力。
2. 新连接器生态。
3. 大图谱重构。
4. 多用户协作。
5. 重型视觉改版。

### 2.4 Phase 0 产出

1. `PHASE_0_MVP_ACCEPTANCE.md` 验收清单。
2. 一条可演示真实用户流。
3. 自动化测试全绿。
4. 阻断级缺陷清零或明确记录。

## 3. 包 B：Phase 1 / V1.1 观点提纯

目标：

```text
让研思录从“能记录、能组织”推进到“能提纯、能判断、能进入知识创作”。
```

### 3.1 第一优先：永久笔记观点提纯面板

先在永久笔记主工作区右侧做 `观点提纯` 面板。

最小必做字段：

1. `thesis`：一句话判断。
2. `three_line_summary`：三句话压缩。

渐进字段：

1. `boundary_or_counterpoint`：边界、反方、不适用条件。
2. `open_questions`：未解问题。

验收标准：

1. 用户能手写保存 `thesis`。
2. 用户能手写保存 `three_line_summary`。
3. 观点状态能影响笔记状态展示。
4. AI 只能给候选，不能直接确认字段。

### 3.2 第二优先：观点提纯队列

做一个轻量观点提纯队列，帮助用户知道下一步最值得补什么。

队列类型：

1. 待一句话判断。
2. 待三句话压缩。
3. 待边界/反方。
4. 待主题归属。

验收标准：

1. 队列来自真实笔记状态，不是静态列表。
2. 点击队列项能回到对应永久笔记。
3. 用户完成提纯后队列状态更新。

### 3.3 第三优先：主题层观点提纯

把 `IndexCard` 从分组对象推进为主题思考对象。

优先字段：

1. `central_question`
2. `thesis`
3. `three_line_summary`

验收标准：

1. 主题能回答“这个主题围绕什么问题形成”。
2. 主题能聚合相关永久笔记。
3. 主题不自动由 AI 创建为完成态。

### 3.4 第四优先：写作意图澄清

写作项目不直接从脚手架开始，而先要求澄清：

1. `intent`：我想说明什么。
2. `desired_reader_takeaway`：我希望读者带走什么判断。
3. 写作前检查：是否有足够永久笔记、中心问题、结构缺口。

验收标准：

1. 写作项目更多从已有判断出发。
2. 脚手架生成前能提示缺口。
3. 反方、边界、待补问题不会被抹平。

### 3.5 AI 候选态原则

AI 可以做：

1. 建议一句话判断。
2. 建议三句话压缩。
3. 提醒边界、反方、不适用条件。
4. 建议中心问题。
5. 提醒关系、重复、冲突、桥接缺口。

AI 不可以做：

1. 自动确认 `thesis`。
2. 自动改写永久笔记正文。
3. 自动创建 confirmed relation。
4. 自动创建正式主题。
5. 自动替用户完成写作判断。

## 5. 成功信号

Phase 0 成功信号：

1. 用户敢把真实资料放进来。
2. 用户知道文件始终归自己所有。
3. 产品不会静默损坏本地内容。
4. 主流程可重复演示。

V1.1 成功信号：

1. 用户愿意高频补全 `thesis`。
2. 用户愿意填写 `three_line_summary`。
3. 用户开始围绕 `central_question` 组织主题。
4. 写作项目从已有判断出发，而不是从空白页开始。
5. AI 像具体位置上的思维陪练，而不是万能代写入口。
