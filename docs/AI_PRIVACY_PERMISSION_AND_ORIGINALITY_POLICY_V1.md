# AI Privacy, Permission, and Originality Policy V1

> Status: draft policy
> Date: 2026-05-10
> Workstream: `feat/ai-agent-layer`
> Related: `CONTEXT_PACK_SCHEMA_V1.md`, `MODEL_ROUTING_POLICY_V1.md`, `AI_ARTIFACT_SCHEMA_V1.md`, `SCHEDULED_AGENT_TASKS_V1.md`

## 1. Purpose

This document defines the policy boundary for what the AI / Agent layer can read, where it can send data, what it can do in the background, and how AI-generated content is kept separate from human-authored notes.

The goal is to make the AI layer useful without eroding user trust or diluting original thinking.

## 2. Core Principles

- Human-authored notes are the user's intellectual property and primary source of truth.
- Agents can suggest, connect, summarize, question, and draft, but they do not silently become the author.
- Privacy policy runs before model routing and cost optimization.
- Background agents have stricter permissions than foreground user-triggered actions.
- AI outputs are stored as artifacts by default.
- Promotion into note content requires explicit user action.
- `local_only` data must never fall back to cloud providers.

## 3. Content Origin Labels

All note blocks, artifacts, and promoted content should use origin labels where possible.

| Origin | Meaning |
| --- | --- |
| `human_authored` | Written directly by the user |
| `ai_generated` | Generated by an agent or model |
| `source_derived` | Derived from external paper, webpage, RSS, PDF, book, or imported source |
| `user_accepted` | AI/source-derived content explicitly accepted by the user |
| `user_rewritten` | AI-assisted content substantially rewritten by the user |
| `system` | Product-generated metadata or system text |

MVP can start at artifact-level origin and later move toward block-level origin.

## 4. Privacy Modes

| Mode | Meaning | Cloud Model Allowed? | Default |
| --- | --- | --- | --- |
| `normal` | Standard note/source material | Yes if workspace policy allows | Default for non-sensitive content |
| `private_project` | Sensitive project or user-marked private scope | Only with user/workspace permission | Requires confirmation by default |
| `local_only` | Must stay local/private | No | Cloud fallback blocked |
| `enterprise_restricted` | Must use approved enterprise providers | Only approved providers | Admin controlled |

Privacy mode should be attached to:

- Notes.
- Source documents.
- Projects.
- Context Packs.
- Scheduled tasks.
- AI artifacts.
- Agent runs.

## 5. Permission Levels

| Permission | Meaning | Background Allowed? | Confirmation Required? |
| --- | --- | --- | --- |
| `read_metadata` | Read titles, tags, project ids, timestamps | Yes | No |
| `read_snippet` | Read short snippets | Yes if privacy allows | Sometimes |
| `read_full_note` | Read full note content | Limited | For private scopes |
| `read_source` | Read imported or external source docs | Yes if source policy allows | No by default |
| `create_artifact` | Create AI artifact | Yes | No |
| `create_suggestion` | Create link/question/draft suggestion | Yes | No |
| `request_confirmation` | Ask user for approval | Yes | N/A |
| `promote_artifact` | Turn artifact into note/link/task | No background | Yes |
| `write_human_note` | Directly mutate user-authored note | No for MVP | Always |

MVP should not expose `write_human_note` to agents.

## 6. Foreground vs Background Rules

Foreground user-triggered runs can do more because the user is present.

Foreground allowed:

- Read current note.
- Read selected notes.
- Use cloud model if privacy mode allows.
- Ask confirmation for expensive or private actions.
- Create artifacts and suggestions.

Background allowed:

- Read metadata.
- Read snippets if privacy allows.
- Read scheduled source scope.
- Create artifacts and suggestions.
- Log run events.

Background not allowed by default:

- Read private full notes.
- Use cloud model for `private_project` without prior permission.
- Use cloud model for `local_only`.
- Promote artifacts into notes.
- Mutate human-authored notes.
- Run broad whole-vault deep synthesis.

## 7. Cloud Model Policy

Before any cloud model call:

1. Build or classify Context Pack privacy mode.
2. Check workspace policy.
3. Check user mode and model pack.
4. Check whether selected provider is allowed.
5. Request confirmation if required.
6. Log decision in Agent Run Log.

Rules:

- `normal`: cloud allowed if workspace policy allows.
- `private_project`: confirmation or workspace policy required.
- `local_only`: cloud blocked.
- `enterprise_restricted`: only approved providers.

Fallback to cloud must never happen silently for private/local scopes.

## 8. Originality Boundary

Agents can create:

- `ResearchCard`
- `LinkSuggestion`
- `ConflictSuggestion`
- `ReflectionPrompt`
- `SynthesisDraft`
- `OutlineDraft`
- `QuestionCard`
- `ProjectDigest`

Agents cannot directly create or overwrite:

- `human_authored` note content.
- Final user claims.
- User-owned conclusions.
- User-approved graph edges.

Promotion path:

```text
AI artifact
  -> user reviews
  -> user accepts, revises, or rewrites
  -> core note app performs final write
  -> user decision is logged
```

## 9. Promotion Rules

Promotion requires:

- Explicit user action.
- Source provenance preserved.
- User decision event recorded.
- Origin updated to `user_accepted` or `user_rewritten`.
- Original AI artifact remains traceable unless user deletes it.

Promotion types:

- Artifact to new note draft.
- Artifact excerpt to existing note.
- LinkSuggestion to graph edge.
- ResearchCard to source note.
- QuestionCard to question/task inbox.

## 10. Source-Derived Claims

When AI outputs factual claims from external sources:

- Preserve source id or URL.
- Keep citation metadata where available.
- Mark content as `source_derived` or `mixed`.
- Do not present source-derived claims as the user's original thought.
- Prefer summaries and links over unsourced assertions.

If a model cannot provide source references for a claim that requires citation, the artifact should be marked lower confidence or require review.

## 11. Sensitive Data Handling

Sensitive data examples:

- API keys and credentials.
- Personal identifiers.
- Private business details.
- Legal/medical/financial records.
- Unpublished manuscripts or confidential research.
- User-marked private projects.

Default behavior:

- Redact credentials and obvious secrets.
- Do not store full sensitive text in Run Logs.
- Prefer references/hashes over raw content.
- Require confirmation before cloud processing private scopes.
- Allow local/private mode for sensitive work.

## 12. User Controls

Users should be able to:

- Pause all background AI.
- Disable cloud AI for a workspace or project.
- Mark notes/projects as local-only.
- Review AI artifacts before promotion.
- Delete AI artifacts.
- Delete or expire run logs, subject to product/enterprise policy.
- Configure BYOK or local/private providers in advanced settings.

Novice controls should stay simple:

- AI mode.
- Privacy mode summary.
- Pause background AI.
- Usage indicator.

## 13. Admin Controls

Team or enterprise admins should be able to:

- Allowlist providers.
- Disable BYOK or require workspace-managed keys.
- Disable cloud fallback.
- Set retention policy.
- Set monthly budgets.
- Require enterprise gateway.
- Export audit logs.
- Define project-level privacy defaults.

## 14. Run Log Requirements

Every run should record:

- Privacy mode.
- Whether cloud model was used.
- Provider and model reference.
- Context Pack id.
- Tool calls.
- Guardrail decisions.
- User confirmation events.
- Created artifact ids.
- Fallback events.

Run logs should not store raw secrets or full private note text by default.

## 15. MVP Policy

MVP should enforce:

- AI outputs are artifacts by default.
- No direct agent write access to human-authored notes.
- Privacy Gate before Model Policy.
- `local_only` cannot use cloud providers.
- Background tasks cannot promote artifacts.
- User confirmation before private cloud fallback.
- Run Log records provider/model/context/artifact references.

MVP can defer:

- Block-level origin tracking.
- Full enterprise policy console.
- Advanced redaction engine.
- Automated sensitive data classifier.
- Fine-grained sharing controls.

## 16. Open Questions

- When exactly does `user_accepted` become `user_rewritten`?
- Should users be able to permanently delete run logs for privacy?
- Should source-derived claims require citation before artifact creation succeeds?
- What is the first version of secret/credential detection?
