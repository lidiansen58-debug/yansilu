## 2. 产品定义

研思录是一款面向学习、研究与写作的本地优先思维工具。
它不应被定义成“更强的知识库”，而应被定义成“把资料转化为思想的工具”。
它既不应让外部摘录替代理解，也不应让 AI 替代理解。

它的目标不是让用户“记得更多”，而是让用户更稳定地完成这条链路：

`资料进入 -> 书摘/原创笔记 -> 连接与图谱 -> 写作项目 -> 脚手架草稿`

它服务的是“思想形成”和“原创表达准备”，而不是通用记录、资料囤积或 AI 代写。
在这个过程中，AI 的角色应是“思想加速镜”，不是“思想替身”。

---

## 3. 目标用户

1. 学习者与研究者：需要把材料整理成可复用观点，而不是只做摘录。
2. 长文写作者：需要把平时积累推进到结构化写作，而不是反复从空白页开始。
3. 知识型内容创作者：需要长期维护主题、证据、连接与写作路径。

---

## 4. 当前 MVP 承诺

当前版本承诺交付的是一个“真实可跑的本地工作流”，不是完整理想态。

当前 MVP 必须成立的事情只有这几类：

1. 用户可以设置/切换本地工作区，并维护目录树。
2. 用户可以创建、编辑、保存、移动三类笔记。
3. 用户可以在原创笔记中建立 `[[wikilink]]`、标签与反向链接关系。
4. 用户可以导入 Markdown/Obsidian 内容，并在确认前看到预览。
5. 用户可以基于原创笔记创建写作项目，并拿到脚手架草稿。
6. 主工作台与搜索结果默认优先强调“形成了多少原创笔记/观点”，而不是“导入了多少资料”。

---

## 5. 明确不做

1. 不做 AI 一键生成原创笔记正文并自动保存。
2. 不做默认输出完整终稿的代写能力。
3. 不做把首页做成全局图谱浏览器。
4. 不做泛化任务管理、协作流、社交流。

---

## 6. 当前版本的关键取舍

## 6.1 本地优先

1. 用户内容以本地 Markdown 为主。
2. SQLite 负责标签、关系、图谱、检索等结构信息。
3. 用户必须始终能拿回自己的文件。

## 6.2 原创优先

1. 原创笔记是写作流程的唯一合法输入。
2. 来源材料不能直接越级变成“系统替你写好的观点正文”。
3. AI 的职责是帮助整理结构，不是替用户完成思想工作。
4. 文献摘录本身不代表理解完成；没有转述或判断线索的文献笔记，不应被视为真正完成。
5. AI 可以帮助发现结构、冲突、缺口与写作路径，但不能直接成为永久笔记正文或终稿正文的默认来源。

## 6.2.1 理解优先

1. 研思录必须反对“摘录即完成”的笔记习惯。
2. 文献笔记的价值不在于保存原文，而在于推动用户形成自己的转述、判断和用途。
3. 系统应持续追问：
   - 这段材料你真正理解成什么？
   - 你为什么要保留它？
   - 它会支持什么判断？
4. 文献笔记还必须保留可生成参考引用的来源信息：标题、作者、年份、容器或来源主体、页码/章节/定位，以及 DOI、ISBN、arXiv、URL 或 PDF 路径等可追溯标识。
5. 缺少转述或关键引用信息的文献笔记，只能停留在草稿/待处理状态，不能进入“记录原创”或完成态。

## 6.2.2 AI 边界

1. AI 可以做：索引建议、连接建议、冲突提示、写作脚手架。
2. AI 不可以做：一键生成永久笔记正文、一键生成终稿。
3. 所有核心 AI 建议都必须可解释、可拒绝、可追溯。

## 6.3 表达优先级

1. 首页首先展示思想形成进展，而不是资料堆积进展。
2. 搜索首先返回可复用观点，其次才返回书摘转述与原始摘录。
3. 导入量、来源量、连接器量属于次级运营信息，不应压过原创笔记形成进度。

## 6.3 结构优先

1. 链接、标签、图谱、写作篮都必须服务于“形成结构”。
2. 任何只会制造噪音的能力，都不应进入当前核心版本。

---

## 7. 发布判断

我们可以把当前版本视为合格 MVP，当且仅当：

1. 用户能从本地工作区启动并稳定保存笔记。
2. 用户能把外部资料通过导入或手工方式带进系统。
3. 用户能围绕原创笔记建立关联，并进入写作项目。
4. 系统输出脚手架草稿，而不是绕过用户生成终稿。
5. 桌面目标版本额外完成原生路径选择、文件定位和真机烟测。

---

## 8. 下一阶段但不阻塞当前发布

1. 语音输入。
2. 完整 i18n。
3. 语义搜索与建议系统。
4. 更强的外部连接器。
5. 更完整的桌面安装与打包体验。

---

## 8.1 导入导出路线图（规划，不等于当前 MVP 承诺）

导入导出的排序不能只看用户量，还要同时看：

1. 用户规模与覆盖面。
2. 用户是否真的有迁移意愿。
3. 外部格式是否开放、稳定、适合机器处理。
4. 是否与研思录的 `Source / Literature Note / Permanent Note` 模型匹配。
5. 实现成本与维护成本是否合理。

基于这些标准，路线图建议如下：

### Phase 1：开放格式底座

1. Markdown 导入。
2. Obsidian vault 导入。
3. Markdown 导出。
4. 附件随导入/导出处理。
5. 导入预览、确认、记录、回滚。

这部分是当前最优先的发布门槛，因为它最贴近本地优先方向，也最容易先做出可被信任的迁移体验。

### Phase 2：主流迁移入口

1. Notion 导入。
2. Readwise 导入。
3. Zotero 导入。

这部分的价值在于：Notion 带来最大规模的迁移入口；Readwise / Zotero 则最贴近“阅读 -> 摘录 -> 原创 -> 写作”的核心链路。

### Phase 3：生态覆盖补齐

1. Evernote 导入。
2. Logseq 导入。
3. Roam 导入。
4. 评估 OneNote / Apple Notes。

这部分属于生态覆盖补齐，不应先于前两阶段成为当前发布门槛。

补充原则：

1. 优先支持“导出文件/导出包导入”，不优先做 API 直连。
2. 优先做单向稳定导入导出，不做双向同步。
3. 优先保留内容与结构，不优先追求样式 100% 还原。
4. 规划中的来源不因为代码里出现实验性实现，就自动升级为当前版本承诺。
5. 新来源进入开发前，仍需通过本文件第 9 节的决策检查清单。

## 9. 决策检查清单

每一个新增需求进入当前版本前，都必须回答这五个问题：

1. 它是否真的帮助用户从资料走向原创笔记？
2. 它是否真的帮助用户从原创笔记走向写作结构？
3. 它是否保留了用户的内容主权？
4. 它是否在放大结构价值，而不是增加界面噪音？
5. 它是否会把产品拉向“泛化笔记工具”或“AI 代写工具”？

如果答案不清晰，就不应进入当前 MVP。
