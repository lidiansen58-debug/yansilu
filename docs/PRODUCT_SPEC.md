# 研思录 Product Spec v1.5

## 1. 文档目的

本文档用于定义研思录当前 MVP 的产品边界、强约束、发布范围与验收口径。

当本文档与旧版规划文档发生冲突时，优先级按以下顺序处理：

1. `PRODUCT_SPEC.md`
2. `ACCEPTANCE_TESTS.md`
3. `IMPLEMENTATION_STATUS.md`
4. 其他历史规划文档

---

## 2. 当前版本的发布目标

当前版本不是“知识管理理想态”的完整实现，而是一个可真实运行的本地优先 MVP。
它的定位不是“更强的知识库”，而是“把资料转化为思想的工具”。
更具体地说，它不只是笔记容器，还应逐步成为观点提纯工作台，帮助用户把复杂想法压缩成清晰观点、中心问题与写作意图。早期文档中的“思想提纯器”属于旧称，后续主称统一为“观点提纯”。

本版本必须让用户能够在自己的本地目录中完成以下闭环：

`随笔/文献/导入素材 -> 永久笔记 -> 关联网络 -> 主题索引 -> 写作项目 -> 脚手架草稿`

从 v1.5 起，产品设计必须明确：

`永久笔记是主对象，随笔笔记和文献笔记只是通向永久笔记的辅助入口。`

这意味着产品不再把随笔、文献、永久笔记设计成三个同等重量的知识库，而是把所有素材流都导向用户自己的原创判断。

发布目标分两层：

1. 先稳定跑通“真实 API + 本地 Vault + Web 原型”。
2. 在此基础上补齐桌面壳集成，达到“desktop-runnable MVP”。

本版本的最终结果是“帮助用户把资料加工成可复用观点，并把观点推进到写作结构”，不是“替用户自动写完整终稿”。

---

## 3. 产品北极星与非目标

### 3.1 北极星

1. 用户能把外部资料、随手想法和阅读材料转成自己的可复用永久笔记。
2. 用户形成了多少永久笔记，比用户导入了多少资料，更能代表产品价值。
3. 用户能围绕永久笔记建立连接、索引和可解释关系，而不是堆积孤立笔记。
4. 用户能看见反方、边界、冲突与概念错位，而不只是收到“相似内容”推荐。
5. 用户能从已有积累进入写作结构，而不是从空白 prompt 开始。
6. 用户应能把永久笔记与主题索引压缩成一句话、三句话与中心问题，而不是停留在“记下来了但说不出来”。
7. 用户始终持有自己的 Markdown 内容与本地目录。
8. 产品应帮助用户拿回判断、组织与写作的主动权，而不是把自己包装成“替你完成一切的外部神力工具”。

### 3.2 非目标

1. 不做通用型笔记大全或项目管理工具。
2. 不做 AI 一键生成永久笔记正文并自动保存。
3. 不做默认生成完整文章终稿的代写产品。
4. 不把首页默认设计成全量全局图谱浏览器。
5. 不把产品主叙事设计成“导入了多少资料、收集了多少条目”的资料仓库故事。
6. 不把“AI 帮你写完”设计成首页、 onboarding 或核心转化路径的主承诺。

## 4. 统一术语

1. `Vault`：工作区根目录；UI 可显示为“目录工作区”或“目录”。
2. `Directory`：工作区中的一个目录节点，带有 `fsPath`、父子关系、隐藏状态与容量配置。
3. `随笔记录 / fleeting note`：快速捕捉线索的素材入口，不是长期知识单元。
4. `书摘笔记 / literature note`：保存来源证据、摘录和转述的证据入口，不是第二套知识库。
5. `永久笔记 / permanent note`：用户基于理解形成的观点型笔记，是研思录的一等知识单元，也是写作篮的唯一合法输入。
6. `索引 / index card`：围绕某个主题、邻近区域、顺序链路或自由连接组织永久笔记的结构对象。
7. `写作项目 / writing project`：围绕一个写作目标组织若干永久笔记的容器。
8. `脚手架草稿 / draft scaffold`：系统生成的结构化写作骨架，包含分节、证据映射与开放问题。
9. `草稿笔记 / draft note`：由用户维护、可绑定到写作项目的永久笔记版本，不等同于系统代写终稿。
10. `本地图 / 主题图 / 写作图`：分别围绕当前目录、某一主题结构、某个写作项目组织出来的三种问题导向图谱视角。

## 5. 范围分层

## 5.1 P0 发布门槛

以下能力属于当前版本必须交付的核心范围：

1. 工作区设置、切换与默认目录初始化。
2. 目录的新建、重命名、移动、隐藏、删除、设置保存路径。
3. 三类底层笔记对象的基础创建、编辑、保存、读取、删除与移动；其中默认主路必须指向永久笔记。
4. 永久笔记中的 `[[wikilink]]`、`#tag`、关联结果、反向链接与同标签检索。
5. 当前目录范围的图谱查看、路径与冲突辅助结果。
6. Obsidian 导入预览与确认导入。
7. Markdown 导出与本地文件/目录定位能力。
8. 写作项目、写作篮、脚手架草稿、脚手架 Markdown 导出/复制、草稿笔记绑定与版本切换。
9. 原创性守卫：覆盖永久笔记保存与导入确认中的永久笔记候选。

## 5.2 P0.5 桌面可运行门槛

以下能力不是浏览器原型的阻塞项，但属于“桌面可运行 MVP”门槛：

1. Tauri 原生目录选择与路径揭示能力可用。
2. 从设置、目录、笔记入口触发的桌面文件命令统一可用。
3. Windows 真机验证通过。
4. macOS / Linux 至少完成基本烟测。
5. 桌面构建前检查与打包前检查可执行。

## 5.3 P1 / P2 延后项

以下能力可以继续做，但不属于当前 MVP 发布门槛：

1. 语音输入。
2. 全量 i18n 与中英切换。
3. 语义搜索、连接建议、索引建议、时间演化图。
4. 默认全局图谱。
5. AI 直接生成完整终稿正文。
6. 更重型的连接器同步能力与企业化协作能力。

## 6. 核心用户流

## 6.1 工作区启动与切换

1. 用户首次进入时可设置工作区根目录。
2. 系统初始化默认目录、SQLite 索引库与缓存目录。
3. 用户后续可切换到另一套 Vault。
4. 切换后，目录树、笔记列表、图谱和写作数据必须同步切换。

## 6.2 录入与形成笔记

1. 用户默认应从永久笔记工作台开始；随笔记录、书摘笔记和导入候选作为素材入口存在。
2. 用户可直接新建永久笔记，也可从随笔、书摘或导入候选记录永久笔记。
3. 书摘笔记必须强调“材料 + 我的转述 + 它可能支持的判断”，完整来源信息通过渐进披露补齐。
4. 系统不得把摘录内容直接落库为永久笔记正文。
5. 书摘笔记与永久笔记必须分层展示，不得在首页、主列表和主搜索结果中被视为同一类对象。
6. 永久笔记在保存前必须经过用户明确确认该表述代表自己的判断，不能把外部摘录或 AI 建议静默落成“已拥有的观点”。
7. 文献笔记没有“转述”时，不得被标记为已转述或可记录永久笔记。
8. 文献笔记编辑器应明确区分“原文/摘录”与“我的转述/理解”，但不应在默认首屏展示所有引用字段。
9. 永久笔记在保存前，如与来源材料或文献摘录高度相似，必须给出明显警示；超阈值时可阻止保存。

## 6.3 建立结构

1. 用户在永久笔记中输入 `[[` 时，应得到候选联想。
2. 标签检索默认覆盖“当前根目录范围”，而不是仅当前已展开的一个子目录。
3. 建立结构的核心对象除了标签与链接，还包括四类索引：主题索引、邻近索引、序列索引、自由连接索引。
4. 关系面板必须能显示引用去向、反向链接、同标签相关结果，以及支持/冲突类关系。
5. 图谱默认从当前目录进入，并提供本地图、主题图、写作图三个主视角；不自动打开全局图。

## 6.4 导入闭环

1. 所有导入必须先预览，再确认写入。
2. 用户可在确认前排除候选项。
3. 导入结果必须展示创建结果、跳过原因与后续整理入口。
4. 当前简化版导入不要求持久历史或回滚入口。

## 6.5 写作闭环

1. 写作不应从空白 prompt 或空白页开始，而应从成熟永久笔记或主题索引进入。
2. 主题索引可以作为组织入口，用来筛入一组永久笔记进入写作篮。
3. 写作篮中的实际证据单元必须是永久笔记，不允许书摘笔记或原始摘录直接进入。
4. 用户可基于若干永久笔记创建写作项目。
5. 系统输出的是脚手架草稿，不是完整终稿。
6. 脚手架必须帮助用户把多层观点组织成整体表达，而不是简单搬运资料。
7. 用户可把脚手架继续绑定到自己的草稿笔记，并保留版本关系。
8. 写作入口应默认来自已有永久笔记、索引、图谱或写作篮，而不是以空白 prompt 或 AI 代写入口作为主起点。

## 7. 功能需求

## 7.1 工作区与目录

1. 默认目录至少包括：
   - `notes/fleeting`
   - `notes/literature`
   - `notes/original`
2. 目录必须持久化以下最小信息：`id`、`name`、`parentId`、`fsPath`、`hidden`、`maxNotes`。
3. 默认目录不可误删，但其下可新增自定义子目录。
4. 目录移动或修改 `fsPath` 时，系统必须同步修正子目录与 Markdown 文件路径映射。
5. 浏览器原型可接受 prompt 降级选路径；桌面发布版必须优先使用原生目录选择器。

## 7.2 笔记类型与编辑

1. 笔记类型由所在根目录决定，避免用户在同一目录内混写不同语义对象。
2. 书摘笔记默认编辑界面只展示四块：
   - 摘录或材料
   - 我的转述
   - 它可能支持的判断
   - 记录永久笔记入口
3. 书摘笔记的完整来源字段必须能支撑未来生成参考引用，但应放入折叠的“来源详情”中：
   - 引用标题
   - 作者
   - 年份
   - 容器信息（书名、期刊、会议、网站或资料库）
   - 出版社或来源主体
   - 页码、章节、段落、时间戳或其它定位信息
   - 版本、译者、编者（有则录入）
   - DOI、ISBN、arXiv、URL 或 PDF 路径等至少一个可追溯标识
   - 摘录
   - 转述
4. 书摘笔记进入“记录永久笔记”前，必须具备用户自己的转述；进入正式写作证据前，再强提示补齐引用标题、作者、年份、定位信息和一个可追溯标识。
5. 永久笔记除正文外，应支持 `boundary_or_counterpoint` 一类字段，用于记录反方、边界条件、反例或不适用范围。
6. 永久笔记应逐步支持 `thesis` 与 `three_line_summary`，用于思想压缩。
7. 永久笔记保存时必须触发原创性守卫。
8. 永久笔记正文中的 `[[标题]]` 必须保持 Markdown 可读，同时在 SQLite 中绑定确定的 `note_id`。
9. 正文中的 `#标签` 必须写回结构化标签索引，并可点击检索。
10. 编辑器必须提供保存反馈、脏状态提示与草稿恢复能力。

## 7.3 首页与搜索优先级

1. 首页/工作台的主价值反馈必须优先回答“形成了多少永久笔记”，而不是“导入了多少资料”。
2. 首页第一层指标应优先显示：
   - 永久笔记总数
   - 最近新增永久笔记
   - 已接入关系网络的永久笔记数
   - 可写主题数
   - 待从素材继续记录永久笔记的条目数
3. 导入条数、来源数、连接器统计只应出现在导入工作区或二级信息位，不应成为首页主 KPI。
4. 搜索结果必须优先展示可复用观点，而不是大段原始摘录。
5. 搜索排序默认优先级应为：
   - Original/PermanentNote 的标题、核心观点、标签
   - 索引笔记与关联的观点型结果
   - LiteratureNote 的转述
   - LiteratureNote 的原始摘录
6. 当书摘笔记与永久笔记命中同一主题时，永久笔记应作为主结果，书摘笔记只作为证据与追溯入口。
7. 搜索摘要默认优先显示观点句、标题或转述，不默认用长摘录占据结果主视觉。

## 7.3.1 AI 建议边界

1. AI 建议必须默认以“建议”形式出现，而不是自动写入核心知识对象。
2. AI 可参与的核心场景包括：索引建议、连接建议、冲突提示、写作脚手架。
3. AI 可辅助一句话压缩、三句话压缩、中心问题提炼与写作意图澄清，但默认只能生成候选建议。
4. AI 不得直接生成并保存永久笔记正文。
5. AI 不得把“完整终稿”设计为写作链路的默认输出。
6. 所有核心 AI 建议都必须满足：
   - 可解释：说明为什么给出这个建议
   - 可拒绝：用户可以显式拒绝，不影响主流程
   - 可追溯：可回到触发该建议的笔记、摘录或关系

## 7.4 索引、关联、标签与图谱

1. 四类索引必须作为独立对象存在，不得退化为纯标签、纯文件夹或隐藏实现细节。
2. 用户必须可以创建和维护四类索引：
   - 主题索引
   - 邻近索引
   - 序列索引
   - 自由连接索引
3. 关系面板至少展示：
   - 当前笔记指向的关联
   - 反向链接
   - 同标签相关笔记
   - 支持关系与冲突关系
4. 标签结果默认按当前根目录过滤，但不应局限于当前已经展开的单个子目录。
5. 图谱 MVP 必须支持本地图、主题图、写作图三种主视角。
6. 图谱默认展示当前目录范围的关系网络，不默认打开全局全量图。
7. 图谱结果必须支持节点回跳、局部聚焦与基础说明信息。
8. 图谱至少应帮助用户识别：
   - 哪些观点互相支持
   - 哪些观点彼此冲突
   - 哪些区域缺少桥接节点，导致主题过渡或写作链路断裂
9. 图谱应提供“冲突/张力视图”，优先暴露重名、孤立观点、待补链接理由，以及其他可能导致概念错位的信号。

## 7.5 导入与导出

1. Obsidian 导入必须支持预览。
2. 预览结果必须区分：
   - 可导入
   - 警告
   - 阻断
   - 已排除
3. 确认导入后必须返回创建结果与跳过原因摘要。
4. 用户必须可以在确认后继续进入后续整理、写作或导出链路。
5. Markdown 导出必须能导出用户内容，并保留可读文件结构。

## 7.6 写作中心

1. 写作项目创建时，输入必须是至少一条永久笔记；主题索引可作为候选入口，但最终必须展开为具体永久笔记。
2. 写作项目至少包含：`title`、`goal`、`audience`、`tone`、`basketNoteIds`；可选记录 `relatedIndexIds` 作为进入写作的主题入口。
3. 写作项目应逐步支持 `intent` 与 `desiredReaderTakeaway`，帮助用户在生成脚手架前先澄清写作意图。
4. 脚手架草稿至少包含：
   - 分节标题
   - 每节/每段用途或意图
   - 证据笔记映射
   - 缺口标注
   - 反方或边界提醒
   - 开放问题/待补问题
5. 开放问题与写作提示应优先覆盖反方观点、边界条件、概念错位与待补漏洞，而不只是顺着相似观点展开。
6. 系统可以导出或复制脚手架 Markdown。
7. 系统可以把某条永久笔记绑定为该写作项目的当前草稿笔记，并保存版本关系。
7. MVP 默认不直接生成完整文章正文。

## 7.7 桌面文件能力

1. 用户应能从界面打开目录在文件管理器中的位置。
2. 用户应能从界面定位当前 Markdown 文件。
3. 桌面壳层的权限失败、路径无效、文件被占用等问题，必须返回可读提示。
4. 路径包含空格、中文字符时必须保持可用。

---

## 8. AI 边界

## 8.1 AI 允许

1. 生成导入预览中的结构化候选结果。
2. 提供原创性风险判断、跳过理由与说明。
3. 生成脚手架草稿、开放问题与结构化写作提示。
4. 辅助发现反方、边界、冲突与概念错位。
5. 辅助连接、标签、索引、图谱或写作篮进入更清晰的结构。
6. 提示支持关系、冲突关系与潜在桥接缺口，但不得替用户静默写入这些判断。

## 8.2 AI 禁止

1. 自动写并保存永久笔记正文。
2. 自动把摘录原文直接转为永久笔记正文。
3. 默认替用户输出完整终稿并直接落库。
4. 省略来源、证据或跳过原因说明。

## 8.3 AI 输出要求

1. 结构化输出必须可校验。
2. 所有关键建议必须可解释。
3. 用户必须能拒绝、回退或改写系统给出的结构结果。

## 8.4 导入导出阶段规划（规划，不直接构成当前发布门槛）

导入导出的优先级排序，统一按以下 5 个维度判断：

1. 用户规模与覆盖面。
2. 迁移意愿是否强。
3. 外部格式是否开放、稳定、适合机器处理。
4. 是否与研思录的 `Source / Literature Note / Permanent Note` 模型匹配。
5. 实现成本与长期维护成本。

阶段建议如下：

### Phase 1：当前发布门槛

1. Obsidian 导入。
2. Markdown 导出。
3. 附件随导入/导出处理。
4. 导入预览与确认。
5. 简化版边界清晰，不承诺历史与回滚。

### Phase 2：主流迁移入口

1. Notion 导入。
2. Readwise 导入。
3. Zotero 导入。

### Phase 3：生态覆盖补齐

1. Evernote 导入。
2. Logseq 导入。
3. Roam 导入。
4. 评估 OneNote / Apple Notes。

额外约束：

1. 规划中的来源不因为代码中出现实验性实现，就自动升级为当前版本承诺。
2. 当前发布门槛仍以 `Obsidian 导入 + Markdown 导出` 为准。
3. 新来源进入主线前，必须先补齐预览、确认与最小可用写入流程设计。
4. 优先支持导出包/导出文件导入，不优先做 API 直连。
5. 当前版本不承诺双向同步与平台专用写回。

如需快速查看当前决策结论，可参考 [IMPORT_EXPORT_DECISION_SUMMARY_V1.md](/E:/Projects/Thinking%20in%20Notes/yansilu/docs/IMPORT_EXPORT_DECISION_SUMMARY_V1.md)。

## 9. 发布门槛

一个版本只有在以下条件同时成立时，才可以被称为“当前 MVP 可发布”：

1. 真实 API + 本地 Vault + Web 原型主流程可跑通。
2. 永久笔记、导入、图谱、写作项目、脚手架流程都有自动化测试覆盖。
3. 导入与导出不会静默破坏用户本地内容。
4. 原创性守卫不会被保存流程绕过。
5. 桌面目标版本还需额外通过原生路径选择、文件揭示和真机烟测。

## 10. 当前版本的成功标准

1. 用户能在本地目录中创建并维护三类底层笔记对象，但默认工作流明确以永久笔记为主。
2. 用户能通过导入或手工整理，把外部资料推进为永久笔记，而不是停留在摘录堆积。
3. 用户能在首页和主要浏览路径上感受到“思想形成”的进展，而不是只看到导入量。
4. 用户能用 `[[wikilink]]`、标签、索引和图谱建立写作相关结构，并发现支持、冲突与桥接缺口。
5. 用户能基于永久笔记创建写作项目并得到脚手架草稿。
6. 用户能在不失去内容主权的前提下，持续把积累推进到写作阶段。
