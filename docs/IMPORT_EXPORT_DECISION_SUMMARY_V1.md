# 导入导出决策摘要（当前版）

更新日期：2026-06-03

Legacy note：这份文档原本记录过更宽的 V1 设想。当前版本已经按“精简且能用”的实现重写，旧的 Markdown 导入、导入历史、回滚和多连接器承诺不再视为当前发布范围。

## 1. 这份文档回答什么

这份文档只回答四件事：

1. 当前版本真正承诺哪些导入导出能力
2. 为什么当前只保留最小可用范围
3. 后续连接器现在处于什么状态
4. 哪些旧设计不该再被当成现状

## 2. 一句话结论

当前版本只把 `Obsidian 导入 + Markdown 导出` 作为发布门槛。

其中导入只保留：

- 选择 Obsidian vault
- 预览
- 确认导入

当前不承诺：

- Markdown 导入
- 导入记录持久化
- 导入回滚
- Readwise / Zotero / Notion 等外部连接器

## 3. 为什么收缩到这个范围

导入导出的真正核心不是“来源越多越好”，而是“当前主路径稳定、可解释、可验证”。

本轮收缩的原因是：

1. 当前最真实的用户迁移入口是 Obsidian vault
2. 预览 -> 确认已经足够支撑安全的一次性导入
3. 导入历史和回滚会显著扩大状态管理、测试和文档复杂度
4. 多连接器承诺会让前后端继续背负大量未完成接口
5. 精简后更容易稳定真实浏览器链路和集成测试

## 4. 当前发布承诺

### 4.1 导入

当前版本承诺支持：

1. Obsidian vault 导入
2. 导入预览
3. 用户确认后写入
4. 附件和嵌入资源随导入处理
5. 导入后结果可进入现有笔记和写作流程

当前版本不承诺：

1. Markdown 目录导入
2. 导入记录长期保存
3. 导入回滚
4. 多连接器统一切换体验

### 4.2 导出

当前版本承诺支持：

1. Markdown 导出
2. 导出结果在本地目录中可定位
3. 导出结构保持可读和可迁移

## 5. 当前不承诺的能力

以下内容不属于当前发布门槛：

1. Notion 导入
2. Readwise 导入
3. Zotero 导入
4. Evernote 导入
5. Logseq 导入
6. Roam 导入
7. OneNote 导入
8. Apple Notes 导入
9. 平台专用导出
10. API 直连同步
11. 双向同步

## 6. 当前路线理解

当前路线不是“所有连接器一起做”，而是：

1. 先把 Obsidian 一次性导入走通
2. 保留 Markdown 导出，确保内容能带走
3. 把其余连接器降为明确的 future work

这意味着：

- 连接器优先级讨论现在不会改变当前发布边界
- 旧文档里出现的 `Notion -> Readwise -> Zotero` 顺序，只能视为未来规划参考

## 7. 当前统一导入心智模型

不管未来是否扩展来源，当前产品心智只有这一套：

1. 选择来源
2. 解析原始内容
3. 生成内部候选对象
4. 预览数量、类型、警告和风险
5. 确认后写入

当前明确不包含：

1. 导入记录持久化
2. 导入回滚
3. 无确认自动写入

## 8. 决策红线

下面这些规则在后续阶段也不应轻易放松：

1. 不允许无预览直接写入
2. 不允许把实验性连接器默认升级为当前发布承诺
3. 不允许为了“支持更多来源”牺牲导入结果可解释性
4. 不允许把双向同步提前塞进当前 MVP 主线

## 9. 现在怎么读旧文档

如果你在仓库其他地方看到下列说法，请按下面方式理解：

- `Markdown 导入`：旧设计或未来规划，不是当前实现
- `ImportRecord`：旧历史模型，不是当前最小实现依赖
- `rollback`：当前不支持
- `Readwise / Zotero / Notion`：后续来源，不是当前发布范围

当前事实请以这些文档为准：

- `README.md`
- `docs/API.md`
- `docs/ACCEPTANCE_TESTS.md`
- `docs/CONNECTORS_MVP_CONTRACT.md`
