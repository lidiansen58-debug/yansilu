# 导入导出决策摘要 V1

## 1. 这份文档解决什么问题

这不是实现文档，也不是接口文档。

它只回答 4 个决策问题：

1. 当前版本到底承诺支持哪些导入导出能力。
2. 后续应该按什么顺序扩展来源。
3. 为什么这样排序。
4. 哪些内容现在明确不承诺。

## 2. 一句话结论

当前版本只把 `Markdown / Obsidian 导入 + Markdown 导出` 作为发布门槛。

后续按以下顺序推进：

1. `Phase 1`：Markdown / Obsidian / Markdown 导出
2. `Phase 2`：Notion / Readwise / Zotero
3. `Phase 3`：Evernote / Logseq / Roam / 评估 OneNote 与 Apple Notes

## 3. 为什么不是只按用户量排序

导入导出的优先级不能只看用户量，还要同时看：

1. 用户规模与覆盖面。
2. 用户迁移意愿是否强。
3. 外部格式是否开放、稳定、适合机器处理。
4. 是否与研思录的 `Source / Literature Note / Permanent Note` 模型匹配。
5. 实现成本与维护成本是否可控。

所以排序逻辑是：

1. 先做开放格式底座。
2. 再做最大规模迁移入口。
3. 再做高价值阅读来源。
4. 最后补复杂生态与存量格式。

## 4. 当前发布承诺

### 4.1 导入

当前版本承诺支持：

1. Markdown 文件夹导入。
2. Obsidian vault 导入。
3. 导入预览。
4. 部分候选排除后再确认。
5. 导入记录。
6. 导入回滚。
7. 附件随导入处理。

### 4.2 导出

当前版本承诺支持：

1. Markdown 导出。
2. 附件随导出处理。
3. 导出结果可在本地目录中定位。
4. 导出结构保持可读、可迁移。

## 5. 当前不承诺

以下内容不作为当前版本发布门槛：

1. Notion 导入。
2. Readwise 导入。
3. Zotero 导入。
4. Evernote 导入。
5. Logseq 导入。
6. Roam 导入。
7. OneNote 导入。
8. Apple Notes 导入。
9. 平台专用导出。
10. API 直连同步。
11. 双向同步。

说明：

1. 即使代码中已有某些实验性连接器实现，也不自动等于产品已经正式承诺支持。
2. 只有进入对应阶段，并补齐统一的预览、确认、记录、回滚流程后，才算正式进入发布范围。

## 6. 路线图

### Phase 1：开放格式底座

范围：

1. Markdown 导入。
2. Obsidian 导入。
3. Markdown 导出。
4. 附件随导入导出处理。

目标：

1. 建立本地优先的迁移底座。
2. 做出可被信任的导入回滚体验。
3. 先满足最稳、最通用、最贴近目标用户的能力。

### Phase 2：主流迁移入口

范围：

1. Notion。
2. Readwise。
3. Zotero。

目标：

1. 打开最大规模迁移入口。
2. 服务阅读摘录型用户。
3. 强化“从资料到永久笔记再到写作”的产品主线。

### Phase 3：生态覆盖补齐

范围：

1. Evernote。
2. Logseq。
3. Roam。
4. 评估 OneNote / Apple Notes。

目标：

1. 补齐存量迁移需求。
2. 扩大生态覆盖。
3. 不让复杂来源拖慢前两阶段主线。

## 7. 用户入口策略

每类来源优先选“最省事且最稳”的方式：

1. Markdown / Obsidian：选择本地文件夹。
2. Notion：上传官方导出 zip。
3. Readwise：先支持导出文件导入，后评估 token 同步。
4. Zotero：导入标准导出文件。
5. Evernote：导入 ENEX。

统一原则：

1. 优先导出文件/导出包导入。
2. 不优先做实时 API 同步。
3. 不优先做双向同步。

## 8. 统一导入契约

不管后续接哪种来源，都必须满足同一套产品心智：

1. 选择来源。
2. 解析原始内容。
3. 生成内部候选对象。
4. 预览数量、类型、风险与警告。
5. 允许部分选择。
6. 确认后写入。
7. 生成导入记录。
8. 必要时执行回滚。

## 9. 决策红线

下面这些规则在后续阶段也不应轻易放松：

1. 不允许无预览直接写入。
2. 不允许把实验性连接器默认升级为发布承诺。
3. 不允许为了平台兼容而破坏本地优先原则。
4. 不允许为了“看起来支持更多来源”而牺牲导入结果可解释性。
5. 不允许把双向同步提前塞进 MVP 主线。

## 10. 评审时只需要拍板的事

讨论时只需要对这几件事给出 yes / no：

1. 是否同意当前发布门槛只看 Phase 1。
2. 是否同意后续优先级为 `Notion -> Readwise -> Zotero`。
3. 是否同意后续来源默认优先做“导出包/导出文件导入”，不优先做 API 直连。
4. 是否同意双向同步明确排除在当前路线之外。
